## 2. The Render-Seam Template-Selection Design

**Problem:** `render_cv` hardcodes `CV_TEMPLATE_REL`. We must add template selection WITHOUT breaking `assemble_application`, `Session::{export_application, render_inputs}`, or the existing render tests (all call the no-arg `render_cv(view)` free fn / `Renderer::render_cv`).

**Design — additive enum + an additive trait method with a default, keep the old fn:**

```rust
// render.rs — new public enum, maps variant → repo-relative template path.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Default, Serialize, Deserialize)]
pub enum CvTemplate {
    #[default] Classic,
    Compact,
    Modern,           // include iff modern.typ ships this cycle; else omit the variant entirely
}
impl CvTemplate {
    pub fn template_rel(self) -> &'static str { /* Classic→classic.typ, Compact→compact.typ, ... */ }
    pub fn parse(s: &str) -> Result<Self, CoreError> { /* "classic"|"compact"|"modern"; else typed err */ }
    pub fn is_multi_column(self) -> bool { matches!(self, CvTemplate::Classic) } // feeds ATS check B
}
```

Thread selection through the trait **additively** (default method preserves every existing impl/caller):

```rust
pub trait Renderer {
    fn render_cv(&self, view: &TailoredView) -> Result<Vec<u8>, CoreError>;          // UNCHANGED
    // NEW — default delegates to Classic so existing impls/tests keep compiling untouched:
    fn render_cv_with_template(&self, view: &TailoredView, t: CvTemplate)
        -> Result<Vec<u8>, CoreError> { let _ = t; self.render_cv(view) }
    fn render_cover_letter(&self, letter: &CoverLetter) -> Result<Vec<u8>, CoreError>; // UNCHANGED
}
```

`CliRenderer` **overrides** `render_cv_with_template` to call `self.compile(t.template_rel(), &json)`, and re-points its own `render_cv` to delegate to `…_with_template(view, CvTemplate::Classic)` (so the Classic path is identical bytes-of-behaviour to today). Add a free fn `render_cv_with_template(view, t)` alongside the existing `render_cv(view)`. **`render_cv(view)` stays as-is** = `render_cv_with_template(view, Classic)` — every current caller and test is untouched.

**Backward-compat guarantees (explicit):**
- All 13 existing render tests + `lib.rs`/`aa-desktop` callers use `render_cv`/`Renderer::render_cv` → still compile and behave identically (Classic).
- `assemble_application` keeps calling `render_cv` (Classic) — out of scope to thread template selection through the §G cover-letter export unless DISCUSS-A3 says otherwise.

**embedded-typst handling (DEFERRAL, documented):** the `embedded` module `include_str!`s `classic.typ` at compile time. The `embedded-typst` feature is **already deferred** (see `ci.yml` comment / `doc/COVERAGE.md` / DISCUSS-RENDER) and does NOT run in CI. **Decision: defer parallel template support in the embedded backend.** Add `include_str!` consts for `compact.typ`(`/modern.typ`) behind `#[cfg(feature="embedded-typst")]` ONLY if trivial; otherwise the `EmbeddedRenderer::render_cv_with_template` default falls back to Classic and we record a one-line deferral note in `render.rs` + `doc/COVERAGE.md`. **The CLI path is the contract for #6** (matches how items 1–5 shipped). This keeps the feature-gated module honest without expanding scope.

**Coverage note:** `CvTemplate::template_rel`/`parse`/`is_multi_column` are pure → each variant must be exercised by a unit test to hold the 100%-of-reachable floor (no pragmas expected).

## 3. EARS Requirement IDs

### A — Templates (`R-TPL-*`)
- **R-TPL-1** — WHEN a caller selects a `CvTemplate`, the Renderer SHALL render the tailored view through that template's `.typ` file, producing a non-empty, structurally-valid PDF.
- **R-TPL-2** — The system SHALL provide at least `Classic` (two-column, existing) and `Compact` (single-column, ATS-friendly) templates; `Modern` is OPTIONAL this cycle.
- **R-TPL-3** — Every CV template SHALL consume the SAME data contract (a `MasterCv`/tailored-view conforming to `doc/schemas/master-cv.schema.json`); no template may require fields outside the schema.
- **R-TPL-4** — Each new template SHALL be CLI-renderable via `typst compile templates/cv/<t>.typ out.pdf --input data=<persona>.cv.json --root .` so CI can smoke it.
- **R-TPL-5** — WHEN no template is selected, the system SHALL default to `Classic` (backward-compatible behaviour identical to pre-#6).
- **R-TPL-6** — WHERE the export flow runs, the UI SHALL let the user select among available templates, and the selection SHALL be threaded to the renderer via a Tauri command parameter.
- **R-TPL-7** — WHEN a template string at the command boundary is unrecognised, the system SHALL surface a typed error (never panic, never silent default).
- **R-TPL-8** — Each new template SHALL be documented in `doc/design/pdf-look.md` (layout, column model, intended use).

### B — ATS-readability checker (`R-ATS-*`)
- **R-ATS-1** — The system SHALL provide a PURE function `ats_report(template: CvTemplate, view: &TailoredView) -> AtsReport` with NO IO and NO PDF parsing (checks are framed over template properties + tailored content).
- **R-ATS-2** — `ats_report` SHALL NEVER mutate the view or any content (read-only; `&TailoredView`).
- **R-ATS-3** — WHEN the chosen template is multi-column (Classic), the report SHALL emit a WARN on the column-reliance check; WHEN single-column (Compact), that check SHALL PASS.
- **R-ATS-4** — The report SHALL flag (WARN) a document whose total rendered content exceeds a deterministic length threshold (overly-long document). *Threshold → DISCUSS-ATS-LEN.*
- **R-ATS-5** — The report SHALL flag (WARN) non-standard section headings against a fixed allow-list. *Allow-list → DISCUSS-ATS-HEAD.*
- **R-ATS-6** — The report SHALL flag (WARN) missing extractable text (e.g. an experience/role with no description-bearing achievements, or an empty document).
- **R-ATS-7** — Each check SHALL be representable as a pinnable coordinate: a stable `check_id` (enum) + `Pass|Warn` status + a human message, in a deterministic order.
- **R-ATS-8** — `ats_report` SHALL be deterministic: identical `(template, view)` → identical `AtsReport`.
- **R-ATS-9** — *(stack-bounded)* WHERE the font stack is the fixed bundled Liberation set, the "unusual font" check SHALL PASS by construction (or be omitted with a documented rationale). *→ DISCUSS-ATS-FONT.*

### C — Keyword-coverage panel (`R-KWC-*`)
- **R-KWC-1** — The system SHALL provide a PURE function `keyword_coverage(view: &TailoredView, job: &NormalizedJob) -> KeywordCoverage` reporting, per keyword, FOUND vs MISSING in the TAILORED view.
- **R-KWC-2** — For each FOUND keyword the report SHALL list WHERE it appears: the contributing evidence ids (skill `evidenceIds` / experience id / achievement id), via `Candidate::matching_evidence_ids` restricted to the tailored view's selected evidence.
- **R-KWC-3** — The report SHALL distinguish must-have vs nice-to-have keyword classes.
- **R-KWC-4** — The function SHALL be computed over the TAILORED view (selected/reordered evidence), distinct from `coverage_report` (over the master CV).
- **R-KWC-5** — The function SHALL be VISIBILITY-ONLY: it SHALL NEVER insert, stuff, reorder, or fabricate keywords or content (`&` borrows; no mutation).
- **R-KWC-6** — WHEN a keyword appears in two or more sections, the report SHALL list all contributing evidence ids (deduped, deterministically ordered).
- **R-KWC-7** — WHEN a must-have keyword is absent from the tailored view, it SHALL appear in the MISSING set with an empty evidence list.
- **R-KWC-8** — `keyword_coverage` SHALL be deterministic.

## 6. DISCUSS items (genuine spec gaps — resolve at step 0 of each capability)

- **DISCUSS-A1 (compact layout).** Confirm `compact.typ` is single-column linear: full-width header → professional summary → skills (proposed: inline grouped lists `Languages: A · B · C`, not the dot-rating sidebar, to maximise ATS text extractability) → experience. Confirm we DON'T need rating dots in compact (dots are image-ish circles → an ATS smell). *Lead recommendation: drop dots in compact; standard headings.*
- **DISCUSS-A2 (modern.typ in or out?).** Brief says `modern.typ` OPTIONAL. Recommendation: ship Compact this cycle (it's the ATS-contrast that capability B's column check needs); defer Modern unless cheap. If deferred, OMIT the `CvTemplate::Modern` variant entirely (don't ship a dead/unreachable variant that breaks the coverage floor).
- **DISCUSS-A3 (cover-letter template selection).** Does template selection apply to the §G cover letter too, or CV-only? Recommendation: **CV-only** for #6 (cover letter stays `classic-letter.typ`); narrower scope, no §G seam churn.
- **DISCUSS-ATS-LEN (R-ATS-4 threshold).** "Overly long" needs a deterministic measure over content (we don't parse the PDF). Recommendation: a fixed cap on a content proxy, e.g. total rendered achievement count across selected experiences > N, or total description char-length > M. Need the N/M values.
- **DISCUSS-ATS-HEAD (R-ATS-5 allow-list).** Our templates emit a FIXED heading vocabulary (classic: "Experience", "Languages", "Skills", "Tools & Technologies", "Platforms & Services"). Since headings are template-controlled (not user data), what does a "non-standard heading" check even range over? Recommendation: either (a) make it a template-property assertion (the chosen template's headings ⊆ a standard allow-list → always Pass for our templates, a guard against future templates), or (b) DROP R-ATS-5 as non-applicable to our fixed-template model. *Pick (a) or (b).*
- **DISCUSS-ATS-FONT (R-ATS-9).** Font stack is the fixed bundled Liberation set for ALL templates → "unusual font" is unreachable by construction. Recommendation: implement as an always-Pass coordinate keyed off the known stack (keeps the check visible/auditable, holds 100% coverage trivially) OR omit with a one-line rationale in `ats.rs`. *Pick one.*
- **DISCUSS-KWC-KEY (R-KWC-1 source).** Does the keyword panel key off `job.requirements` (must/nice, what `coverage_report` uses) or the separate `job.keywords` field (currently unused)? The brief says "must-have/nice-to-have keywords" → recommendation: **key off `requirements.must_have`/`nice_to_have`** (consistent with coverage; `job.keywords` from jobparse is sparse/unused). Confirm.
- **DISCUSS-KWC-SEL (R-KWC-2 scope).** `matching_evidence_ids` scans the whole `view.cv`. The tailored view ALREADY drops hidden experiences and caps achievements per role, AND `selected_ids` is available. Do we additionally intersect found evidence ids with `view.selected_ids` to report ONLY selected-evidence locations? Recommendation: **yes, intersect** — the panel is about the tailored output, so a keyword found only in a dropped bullet should read as not-surfaced. Confirm (cheap either way).
